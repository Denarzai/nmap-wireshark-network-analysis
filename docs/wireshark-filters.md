# Wireshark display-filter cheat-sheet

Display filters used to isolate the traffic generated by each scan. These are
**display** filters (the bar at the top of Wireshark), not capture filters.

| Filter | Shows | Why it matters |
|---|---|---|
| `icmp` | Echo Request / Echo Reply | Host-discovery (ping) traffic |
| `arp` | ARP requests / replies | Layer-2 sweeping when ICMP is blocked |
| `tcp.flags.syn==1` | All SYN packets | The outgoing probes of a SYN scan |
| `tcp.flags.syn==1 && tcp.flags.ack==1` | SYN/ACK packets | Replies from **open** ports |
| `tcp.flags.reset==1` | RST packets | Replies from **closed** ports |
| `tcp.port==445` | Traffic on a specific port | Focus on one service (e.g. SMB) |
| `smb \|\| smb2` | SMB protocol | Share/host enumeration |
| `http` | HTTP requests/responses | Web service interaction |
| `dns` | DNS queries/responses | Name-resolution activity |

### Notes / corrections

- The SYN-scan response filter is `tcp.flags.syn==1 && tcp.flags.ack==1` — there
  is **no** `tcp.flags.synack` field in Wireshark. SYN and ACK are separate flag
  bits and must be tested individually.
- Use `==` for equality (`tcp.flags.syn==1`), not a single `=`.
- Combine conditions with `&&` (and), `||` (or), `!` (not), e.g.
  `icmp || arp` to watch both discovery methods at once.

### Reading a SYN scan in the capture

1. Many `SYN` packets leave the attacker toward many destination ports in a
   short window.
2. **Open** ports answer with `SYN, ACK`; the attacker replies `RST` to tear the
   half-open connection down (never completing the handshake).
3. **Closed** ports answer with `RST` immediately.

That fan-out of SYNs to sequential ports is the classic scan signature an IDS
flags.
